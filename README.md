# PPOkemon

Pokemon Showdown 배틀 환경을 기반으로 PPO(Proximal Policy Optimization) 알고리즘을 통해 배틀 전략을 학습하는 강화학습 프로젝트입니다.
Stable-Baselines3의 MaskablePPO를 사용하며, 직접 작성하는 코드는 모두 이 폴더에 위치합니다.

---

## 제한 환경

전체 포켓몬 풀을 사용하면 학습 시간과 자원이 부족하므로 다음과 같이 제한합니다.

| 항목 | 제한 |
|------|------|
| 배틀 형식 | 3 vs 3 |
| 기술 | 공격기만 허용 (변화기 제외) |
| 아이템 | 사용 안 함 |
| 특성 | 배틀에 영향을 주지 않는 특성만 사용 |
| 랭크업 변화 | State에 포함하지 않음 |

포켓몬 수는 단계별로 확장됩니다 (아래 커리큘럼 학습 참조).

---

## 커리큘럼 학습 (3단계)

State 벡터를 처음부터 최종 크기(168차원)로 고정하고, 아직 사용하지 않는 슬롯은 `0.0`으로 채웁니다.
이를 통해 단계 전환 시 SB3 `PPO.load()`로 가중치를 그대로 이어받을 수 있습니다.

| 단계 | 포켓몬 풀 | 타입 구성 | 학습 목표 |
|------|-----------|-----------|-----------|
| Phase 1 | 아르세우스 9종 (단일 타입 9가지) | 9개 타입 | 기초 타입 상성 + 교체 타이밍 |
| Phase 2 | 아르세우스 18종 (단일 타입 18가지) | 18개 타입 | 전체 타입 상성 + 기술 고려 |
| Phase 3 | 실제 포켓몬 18종 (단일 타입) | 18개 타입 | 실수치 스탯 기반 전략, 자기대전(self-play) |

## State (관측 공간) 정의

### 개요

`envs/spaces.py`에 정의된 **168차원 float32 벡터**를 사용합니다. 값의 범위는 `[-1.0, 1.0]`입니다.

### 벡터 레이아웃

```
[0   :120]  내 팀
[120 :240]  상대 팀
```

각 팀은 동일한 구조로 이루어집니다.

```
팀 (84차원)
├── 액티브 슬롯  (36차원)  [0:36]
├── 벤치 슬롯 #1 (24차원)  [36:60]
└── 벤치 슬롯 #2 (24차원)  [60:84]
```

### 액티브 슬롯 (36차원)

| 요소 | 차원 | 범위 | 설명 |
|------|------|------|------|
| HP 비율 | 1 | [0, 1] | `current_hp / max_hp` |
| 포켓몬 ID | 18 | {0, 1} | 최대 18마리 풀 내 인덱스 one-hot |
| 실수치 스탯 | 5 | [-1, 1] | `log(내 스탯 / 상대 액티브 스탯) / log(400)` |
| 기술 #0~#3 | 3×4 | [0, 1] | 기술당: 위력(1) + 특수여부(1) + PP비율(1) |

### 벤치 슬롯 (24차원)

기술 정보와 랭크업 변화는 포함하지 않습니다.
기절·부재 시 슬롯 전체가 `0` 벡터입니다.

| 요소 | 차원 | 범위 | 설명 |
|------|------|------|------|
| HP 비율 | 1 | [0, 1] | 기절 시 0 → 슬롯 전체가 0 벡터 |
| 포켓몬 ID | 18 | {0, 1} | one-hot |
| 실수치 스탯 | 5 | [-1, 1] | `log(내 스탯 / 상대 액티브 스탯) / log(400)` |

### 전체 구조 요약

```
인덱스         내용
──────────────────────────────────────────────────────
[0   :  1]   내 액티브 HP 비율
[1   : 19]   내 액티브 포켓몬 ID (one-hot 18)
[19  : 24]   내 액티브 실수치 스탯 (5, log-relative vs 상대 액티브)
[24  : 36]   내 액티브 기술 #0~#3 (3×4)
[36  : 60]   내 벤치 #1 (hp + id + stats, log-relative vs 상대 액티브)
[60  : 84]   내 벤치 #2 (hp + id + stats, log-relative vs 상대 액티브)
[84  :120]   상대 액티브 슬롯 (같은 구조, 미공개 기술은 0)
[120 :144]   상대 벤치 #1 (log-relative vs 내 액티브)
[144 :168]   상대 벤치 #2 (log-relative vs 내 액티브)
```

### 설계 결정 사항

- **Zero-padding**: Phase 1부터 240차원으로 고정. 미사용 슬롯은 `0.0`으로 채워 가중치 연속성을 보장합니다.
- **타입 인코딩 제외**: 포켓몬 ID one-hot으로 타입 상성을 경험적으로 학습. 명시적 타입 벡터 불필요.
- **상태이상 제외**: 현재 학습 단계에서 상태이상 기술을 사용하지 않으므로 State에서 제외합니다.
- **랭크업 변화 제외**: 현재 로드맵에서 변화기를 사용하지 않으므로 State에서 제외합니다.
- **기술 PP 비율 포함**: `current_pp / max_pp`로 PP 고갈 상황을 행동 결정에 반영합니다.
- **벤치에 기술 미포함**: 교체 후에야 기술이 의사결정에 사용되므로 불필요합니다.
- **실수치 스탯 log-relative 인코딩**: `log(내 스탯 / 상대 액티브 스탯) / log(400)`으로 인코딩합니다. 포켓몬 대미지 공식이 스탯 비율에 비례하므로 log 스케일이 선형보다 학습 신호를 더 명확하게 표현합니다. 상대 정보 미확인 시 zero로 처리합니다.

## Action (행동 공간) 정의

### 개요

`envs/spaces.py`에 정의된 **`Discrete(6)`** 이산 행동 공간을 사용합니다.

3v3 배틀에서 한 턴에 선택할 수 있는 행동은 **기술 사용** 또는 **포켓몬 교체** 둘 중 하나입니다.

```
기술 4개  +  교체 대상 2마리  =  총 6개  →  Discrete(6)
```

### 행동 인덱스 레이아웃

| 인덱스 | 슬롯 | 설명 |
|--------|------|------|
| 0 | move slot 0 | active 포켓몬의 첫 번째 기술 사용 |
| 1 | move slot 1 | active 포켓몬의 두 번째 기술 사용 |
| 2 | move slot 2 | active 포켓몬의 세 번째 기술 사용 |
| 3 | move slot 3 | active 포켓몬의 네 번째 기술 사용 |
| 4 | switch slot 0 | 벤치 포켓몬 #0으로 교체 |
| 5 | switch slot 1 | 벤치 포켓몬 #1으로 교체 |

### Action Masking

매 턴 실제로 사용 가능한 action은 6개보다 적을 수 있으므로, PPO에게 불법 행동을 차단하는 boolean 마스크를 함께 제공합니다.

| 상황 | 마스킹 결과 |
|------|-------------|
| 기술이 3개뿐인 포켓몬 | `mask[3] = False` |
| 벤치 포켓몬 1마리 기절 | `mask[5] = False` |
| 강제 교체 턴 (active 기절 직후) | `mask[0~3]` 전부 `False` |

`mask=False`인 슬롯에는 PPO의 softmax 직전에 `-inf`를 더해 선택 확률을 0으로 만듭니다.

**강제 교체 턴**: active 포켓몬이 기절했을 때 발생합니다. 이 턴에는 기술 사용이 불가능하며 교체만 허용됩니다. `battle.force_switch` 플래그로 구분합니다.

### 주요 함수

| 함수 | 역할 |
|------|------|
| `build_observation(battle)` | 현재 배틀 상태 → 168차원 float32 벡터 반환 |
| `build_action_mask(battle)` | 현재 턴의 유효 행동 마스크 `(6,) bool` 반환 |
| `action_to_order(action, battle)` | 정수 action → poke-env `BattleOrder` 변환 |
| `safe_action(action, battle)` | 마스크 위반 시 유효 행동으로 폴백 (디버깅 전용) |

### 설계 결정 사항

- **`Discrete(6)` 선택**: `MultiBinary`나 `Dict` 공간 대신 단순 정수 공간을 사용하여 `MaskablePPO`와의 호환성을 유지합니다.
- **기술 슬롯 고정 (4개)**: 실제 기술 수가 4개 미만인 경우 masking으로 처리하며, 공간 크기는 항상 6으로 고정합니다.
- **교체 슬롯 2개**: 3v3에서 벤치 크기는 2이므로 switch slot은 2개입니다.
- **`force_switch` 분리 처리**: 강제 교체 턴과 일반 턴을 명시적으로 구분하여 불법 action 선택 가능성을 차단합니다.
